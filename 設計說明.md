# 教練課後復盤與 KPI 回報系統 — 設計說明

> 證據式教練復盤系統。核心不是記學生，而是讓教練復盤「這堂課自己教得好不好」，
> 並讓館長透過後台追蹤每位教練的教學品質與態度表現。
>
> 第一階段：純前端單檔 `index.html` + LocalStorage，打開即用、免部署。

---

## 1. 修改後的頁面區塊架構

系統用「身分」切換功能，登入時選 **教練** 或 **館長**。

### 教練端（填寫，求快）
1. 📋 基本資訊
2. 🎯 今日課程復盤（原定 vs 實際）
3. 📈 教學 KPI（9 項）
4. 🫡 態度 KPI（6 項）
5. 🚩 態度紅旗
6. 🔧 教學方法與環節（有效／無效／安全疑慮）
7. 🪞 教練自我復盤（最需改進／只改一件事／追蹤指標／需要協助）
8. 📊 本堂分數摘要（即時計算）
9. 🤖 AI 復盤分析（按鈕產生）
10. 💬 LINE 群組回報（按鈕產生，一鍵複製）

### 館長後台（看，求快速抓問題）
1. 🏠 館長首頁儀表板
2. 📋 紀錄列表（多條件篩選＋展開看完整復盤）
3. 👤 教練個人統計
4. 🏫 班別統計
5. 🚨 風險提醒
6. ⚙️ 設定（教練/班別名單、AI 模式、資料匯出入）

**設計取捨**：為了「課後時間很短要快填」，KPI 的「原因／證據／改善」三欄**只有打 3 分以下才展開且必填**，5/4 分不必多寫字。這同時達成「證據式」與「快速」兩個目標。

## 2. 每個欄位的名稱與用途

### 基本資訊
| 欄位 | key | 用途 |
|---|---|---|
| 授課教練 | coach | 對應 coaches；統計主鍵 |
| 身分 | role | 主教練/助教/實習教練/館長 |
| 上課日期 | date | 篩選、週/月統計 |
| 星期 | weekday | 由日期自動帶出 |
| 學生人數 | studentCount | 控場難度參考 |
| 班別 | classType | 對應 class_types |
| 開始/結束時間 | start / end | 課堂長度 |

### 今日課程復盤
| 欄位 | key | 用途 |
|---|---|---|
| 今日原定目標 | goal | 復盤的「原定」基準 |
| 實際完成內容 | actual | 復盤的「實際」 |
| 沒達成原因 | notDone | 缺口分析、AI 主要問題來源 |
| 學生最常出現的問題 | commonErr | 判斷是教法問題還是個別問題 |

### 教學 KPI（goal / clarity / demo / command / pace / control / interact / correct / safety）
目標達成、教學清楚度、示範品質、口令與指令、課程節奏、班級控場、學生互動、個別修正、安全管理。各 1–5 分，附原因／課堂證據／下次改善。

### 態度 KPI（punctual / spirit / observe / etiquette / responsibility / reporting）
準時與課前準備、精神投入、專注觀察學生、禮儀與語氣、責任感、課後回報完成度。各 1–5 分。

### 紅旗（多選）
late 遲到、noprep 課前未準備、phone 上課滑手機、chat 上課聊天分心、lowspirit 精神不佳、tone 口氣不當、ignoresafety 忽略學生安全、impatient 對學生缺乏耐心、noobserve 未主動觀察學生、noreport 未完成課後回報、none 無。
勾任一紅旗 → 必填 flagReason / flagEvidence / flagFix / flagTrack。

### 方法與環節
effective 有效教學方法、ineffective 無效教學方法、badPart 節奏/秩序不好的環節、safety 安全疑慮開關 + safetyNote 說明。

### 自我復盤
selfImprove 今日最需改進、oneThing 下次只改一件事、track 下次追蹤指標、help 需要協助開關 + helpNote。

## 3. KPI 計算方式

```
教學 KPI 平均 = 9 項教學分數平均（四捨五入到小數 1 位）
態度 KPI 平均 = 6 項態度分數平均（四捨五入到小數 1 位）
```

等級（顯示用）：
| 平均 | 等級 |
|---|---|
| ≥4.5 | A 優異 |
| ≥4.0 | B 良好 |
| ≥3.5 | C 達標 |
| ≥3.0 | D 待加強 |
| <3.0 | E 需介入 |

分數定義（評分時 hover 各分數按鈕會顯示）：
- 5 表現穩定，能帶動學生狀態，也符合道館標準
- 4 整體良好，只有小地方可調整
- 3 基本達標，但有明顯可改善處
- 2 多次出現問題，已影響課程品質或學生狀態
- 1 嚴重影響課程品質、學生安全、家長觀感或道館形象

## 4. 態度紅旗邏輯

- 勾「無」→ 清空其他所有紅旗。
- 勾任一其他紅旗 → 自動取消「無」。
- 只要有任一真實紅旗 → 顯示並**必填**：發生原因、課堂證據、改善要求、下次追蹤方式（送出前驗證，缺一不可送）。
- 任一真實紅旗 → 該筆紀錄在後台自動列入風險（紅色邊框 + 🚩 標籤）。

## 5. AI 復盤分析規則

預設用**本機規則引擎**（免金鑰、免網路），輸入：教學/態度分數、低分項的原因與證據、紅旗、有效/無效方法、安全疑慮、下次只改一件事。輸出固定六段：

1. **今日主要問題**：依序取「沒達成原因 → 最低的 1–2 個教學項（帶課堂證據）→ 安全疑慮 → 無效方法」。
2. **可能原因**：取低分項填的「原因」＋學生重複錯誤。
3. **具體改善建議**：取低分項填的「下次改善」＋教練自己寫的「只改一件事」。
4. **下次只追蹤一個指標**：用 track 欄；沒填則由 oneThing 推導。
5. **是否需要館長/資深教練介入**：命中任一條件即「需要」並列出原因（需協助、紅旗、安全疑慮、教學均<3.5、態度均<3.5、單項<3）。
6. **給教練的一句話**：依情境挑（有紅旗→講態度；有安全→講安全；態度低→講帶動；教學低→講聚焦一件事；全好→講複製成功）。

> 文案刻意寫成道館口吻、務實可執行，不產生空泛建議；缺證據時直接點名「請補上原因以利追蹤」。
> 進階：在設定填 Anthropic API Key 可改接 Claude 產生更口語的版本（key 只存本機）。

## 6. LINE 回報格式

只放館長一眼要看的：教練、日期、班別、人數、教學均、態度均、今日主要問題、下次只改一件事、態度紅旗、是否需要館長協助、下次追蹤。範例：

```
教練：王教練
日期：2026/06/04（週四）
班別：兒童班
人數：18 人

教學 KPI：4.0 / 5
態度 KPI：4.5 / 5

今日主要問題：
教學清楚度（3分）：同一錯誤在多位學生重複出現

下次只改一件事：
示範後先做3分鐘定格收腳練習，再進入連續踢擊

態度紅旗：無
館長協助：不需要
下次追蹤：學生3次提醒內正確收腳
```

## 7. 後台頁面架構

- **館長首頁**：今日已回報數、本週堂數、本週教學均、本週態度均、本週風險數 → 下接「需處理風險紀錄／需協助教練／最近紅旗／本週改善追蹤事項」。
- **紀錄列表**：日期區間、教練、班別、KPI 分數（均<3.5／單項<3／均≥4）、是否有紅旗、是否需協助 六種篩選；點任一筆展開完整復盤。
- **教練統計／班別統計／風險提醒**：見第 8 點。

---

## 8. 後台統計與風險提醒邏輯

### 教練個人統計
本週/本月堂數、教學均、態度均、最常低分項（≤3 次數排序）、最常紅旗、回報完成度（reporting KPI 平均近似）、連續低分提醒、是否建議培訓/面談（教學均<3.5 或 態度均<3.5 或 連續低分 或 同一紅旗≥2 次）。

### 班別統計
各班教學均、態度均、控場問題次數（control<3）、需館長協助次數；控場問題≥2 次提示加派助教。

### 風險自動標記（單筆紀錄）
- 任一 KPI < 3 分
- 教學均 < 3.5
- 態度均 < 3.5
- 出現態度紅旗
- 有安全疑慮
- 明確需要館長協助
- 同一教練連續 2 次出現相同低分項（跨紀錄比對）

> 「一週內多次未完成回報」在 GAS/雲端階段最準（需排課表）；前端原型先用 reporting KPI 與 noreport 紅旗近似。

## 9. 資料儲存結構

第一階段 LocalStorage 用三個 key：`tkd_coaches`、`tkd_classes`、`tkd_reviews`（每筆 review 內嵌 KPI 分數/明細/紅旗/AI 結果/LINE 文字）。
升級雲端時拆成下列正規化資料表：

| 表 | 主要欄位 | 用途 |
|---|---|---|
| **coaches** | id, name, role(主教練/助教/實習教練/館長), active | 教練名單 |
| **class_types** | id, name(幼幼/兒童/週六/社團/選手/品勢/黑帶/成人), default_coach | 班別 |
| **session_reviews** | id, coach_id, class_type_id, date, weekday, student_count, start, end, goal, actual, not_done, common_err, effective, ineffective, bad_part, safety, safety_note, self_improve, one_thing, track, need_help, help_note, teach_avg, att_avg, created_at | 一堂課一筆，復盤主表 |
| **teaching_kpi_scores** | id, review_id, kpi_key, score(1-5), reason, evidence, fix | 教學 9 項，一筆一項 |
| **attitude_kpi_scores** | id, review_id, kpi_key, score(1-5), reason, evidence, fix | 態度 6 項 |
| **attitude_flags** | id, review_id, flag_key, reason, evidence, fix, track | 該堂勾選的紅旗（none 不存） |
| **ai_review_results** | id, review_id, problems, causes, fixes, track_metric, intervene(bool), intervene_reasons, reminder, engine(rule/claude), created_at | AI 分析結果 |
| **line_reports** | id, review_id, content, sent(bool), sent_at | LINE 回報文字與發送狀態 |

> 升級順序建議：LocalStorage → Google Sheets(GAS Web App) → Supabase/PostgreSQL。
> Supabase 對應上表即一張一張 table，session_reviews 用 RLS 讓教練只看自己、館長看全部。

## 11. 可實作的程式碼修改建議（升級路徑）

**現況**：`index.html` 已是完整可跑原型。資料層集中在檔案開頭 `LS` / `load` / `save` / `getReviews` / `setReviews`。

**接 Google Sheets（GAS）**：
1. 新建一個 GAS Web App，doPost 接 review JSON 寫入 `session_reviews` 試算表分頁，doGet 回傳全部紀錄。
2. 前端只改三個函式：`setReviews()` 改成 `fetch(GAS_URL, {method:'POST', body})`、`getReviews()` 改成快取 + `fetch(GAS_URL)`。其餘 UI/統計邏輯完全不動。
3. 部署雷請參考你既有的 `gas-webapp-gotchas` 筆記（單檔內嵌、& 跳脫、改版要新版本號）。

**接 Supabase**：把 `store.js` 換成 supabase-js，依第 9 點建 8 張表；AI 分析可改用 Edge Function 呼叫 Claude。

**接 Claude AI 分析**：把 `ruleEngineAI()` 旁加一個 `claudeAI(review)`，當設定有 API key 時改呼叫 `claude-opus-4-8`，prompt 帶入分數/證據/紅旗，要求輸出第 5 點的六段格式。建議走後端代理，別把 key 放前端正式環境。
